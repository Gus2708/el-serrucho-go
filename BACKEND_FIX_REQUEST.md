## 📋 TL;DR

Después del despliegue del widget Python V2 (introducción de `id_unico`, `metodo_pago`, `total_bruto`, `created_at` real), tres regresiones rompieron la integridad de datos en Supabase. La app móvil aplicó parches temporales en SQL views, **pero el bug raíz está en el widget Python** y debe arreglarse ahí. Mientras no se arregle, los datos que llegan a la nube siguen siendo inconsistentes.

| # | Problema | Impacto | Tabla afectada |
|---|---|---|---|
| 1 | `productos.costo`, `productos.precio_venta`, `productos.existencia` se escriben como `0.0000` | 7,212 productos sin precio/stock — toda la app de inventario inservible | `productos` |
| 2 | `ventas.total_neto`, `ventas.total_impuesto` y `ventas_detalle.precio_venta` se suben en **VES (bolívares)** en vez de USD | Ingresos/ganancia/ticket en cifras millonarias, gráficos absurdos | `ventas`, `ventas_detalle` |
| 3 | `ventas.total_bruto` siempre se queda en `0` | Cálculo de IVA roto en el modal de detalle | `ventas` |
| 4 | `ventas_detalle.venta_id` se dejó en `NULL` para 52,960 filas (99.8% del detalle) | JOIN clásico FK roto; tuvimos que cruzar por `documento` | `ventas_detalle` |
| 5 | Posibles duplicados de ventas (ej. 25189/25190 con totales idénticos) | Conteo de facturas inflado, ingreso doblado | `ventas` |

## 🌍 Contexto del sistema (para agentes nuevos)

**Stack:**
- Hardware store en Caracas, Venezuela (Ferretería El Serrucho)
- POS de escritorio: **HybridLite** que escribe archivos `.dat` en `H:\` (Windows local)
- Backend sincronizador: **widget Python** (`app.py`, `monitor.py`, `sync.py`, `sync_ventas.py`, `remote_listener.py`) corriendo en una PC en la ferretería
- Cloud DB: **Supabase** (Postgres) — el widget escribe usando la `ANON_KEY` (exposed in plain config)
- Cliente: app móvil React Native + Expo (admin/empleados consultan la data desde sus celulares)

**Reglas de negocio CRÍTICAS:**
- 🇻🇪 **Toda la UI de la app móvil muestra USD ($)**. Nunca bolívares (VES). Nunca tasa de cambio dual.
- HybridLite registra montos en VES con tasa por factura (`THT_FACTORREFERENCIAL`). El widget Python convierte a USD antes de subir a Supabase.
- IVA = 16% incluido en `productos.precio_venta`. `productos.costo` viene **sin** IVA.
- Solo ventas con `status = 1` se muestran (las demás están anuladas).
- `productos`, `ventas`, `ventas_detalle`, `clientes`, `tazas` son **read-only** para la app — el widget Python es dueño exclusivo de las escrituras.

## 🐛 Bug #1 — `productos` con todos los valores en CERO

### Evidencia (verificada hoy 2026-05-08)

```sql
SELECT
  COUNT(*) FILTER (WHERE precio_venta > 0)              AS con_precio,
  COUNT(*) FILTER (WHERE precio_venta = 0 OR precio_venta IS NULL) AS sin_precio,
  MAX(actualizado_en) AS ultimo_sync
FROM productos;
-- Resultado:
-- con_precio: 0
-- sin_precio: 7212
-- ultimo_sync: 2026-05-09 02:08:23 UTC   (el widget SI corrió hace minutos)
```

**Síntoma:** los 7,212 productos tienen `costo`, `precio_venta`, `existencia` todos en `0.0000`. Pero `actualizado_en` está reciente, así que el widget SÍ escribió las filas — solo que con valores en cero.

### Hipótesis de causa raíz (revisar)

1. **Mapeo de columnas roto.** En la versión V2 alguien tal vez renombró las columnas del DataFrame `pandas` que se manda al `client.table('productos').upsert(...)` y los campos numéricos están cayendo en `None` y se serializan a `0`.
2. **Cast a numeric falla silenciosamente** cuando viene texto del `.dat` con formato venezolano (`"1.500,50"`) y se está usando `float()` directo.
3. **Selecciones cambiadas en HybridLite**: tal vez el `.dat` actual tiene los campos `Costo` / `PrecioVenta` / `Existencia` en posiciones distintas y el lector está leyendo otras columnas.

### Acción requerida en `sync.py` (o el archivo equivalente que sincroniza productos)

1. **Agregar logging de validación al final de cada batch:**
   ```python
   import logging
   logger = logging.getLogger(__name__)

   batch = [...]  # lista de dicts antes del upsert

   non_zero_prices = sum(1 for r in batch if (r.get('precio_venta') or 0) > 0)
   if non_zero_prices < len(batch) * 0.5:
       logger.error(
           "ABORT: %d/%d productos con precio_venta=0 — probable lectura mal del .dat",
           len(batch) - non_zero_prices, len(batch)
       )
       raise RuntimeError("Demasiados productos con precio cero, abortando upsert")
   ```
2. **Verificar el parser del `.dat`**: imprimir las primeras 3 filas leídas y comparar contra el archivo en HybridLite. Confirmar que las columnas `Costo`, `PrecioVenta`, `Existencia` están alineadas.
3. **Helper de cast robusto** para texto venezolano:
   ```python
   def safe_decimal(value: str | None) -> float:
       """Convierte '1.500,50' o 'Bs.2.00' o '' a float. Nunca devuelve None."""
       if value is None or str(value).strip() == '':
           return 0.0
       s = str(value).replace('Bs.', '').replace(' ', '').strip()
       # Manejar formato venezolano "1.500,50" → 1500.50
       if ',' in s and '.' in s:
           s = s.replace('.', '').replace(',', '.')
       elif ',' in s:
           s = s.replace(',', '.')
       try:
           return float(s)
       except ValueError:
           logger.warning("safe_decimal: no se pudo parsear %r, devuelvo 0", value)
           return 0.0
   ```

### Verificación post-fix

```sql

       COUNT(*) FILTER (WHERE existencia > 0)   AS con_stock,

   def venta_to_record(row_from_dat: dict) -> dict:
       factor = row_from_dat.get('THT_FACTORREFERENCIAL')
       if not factor or factor <= 0:
           # Fallback: tomar la tasa BCV de tazas, pero loggear advertencia
           logger.warning("Factura %s sin THT_FACTORREFERENCIAL, usando bcv_usd actual",
                          row_from_dat.get('THT_DOCUMENTO'))

       total_neto_ves    = safe_decimal(row_from_dat['THT_MONTOTOTAL'])

       return {
           'id_unico':       int(row_from_dat['THT_IDÚNICO']),  # o como se llame
           'documento':      str(row_from_dat['THT_DOCUMENTO']).zfill(8),
           'fecha_emision':  row_from_dat['THT_FECHAEMISION'],
           'rif_cliente':    row_from_dat.get('THT_RIFCLIENTE'),
           # 🔑 CONVERSIÓN VES → USD usando el factor histórico de la factura
           'total_neto':     round(total_neto_ves    / factor, 4),
           'total_bruto':    round(total_bruto_ves   / factor, 4),  # NUEVO V2
           'total_impuesto': round(total_imp_ves     / factor, 4),
           'metodo_pago':    str(row_from_dat.get('THT_METODOPAGO', '')).strip(),
           'status':         int(row_from_dat.get('THT_STATUS', 1)),
           'numero_control': row_from_dat.get('THT_NUMEROCONTROL'),
           # ⚡ V2: hora REAL de la factura (UTC-4 → UTC)
           'created_at':     localize_caracas(row_from_dat['THT_FECHAHORA']).isoformat(),
       }
   ```

## 🐛 Bug #4 — `ventas_detalle.venta_id` huérfano (52,960 filas)

### Evidencia

```sql
SELECT
  COUNT(*) FILTER (WHERE venta_id IS NULL)     AS sin_fk,
  COUNT(*) FILTER (WHERE venta_id IS NOT NULL) AS con_fk
FROM ventas_detalle;
-- sin_fk: 52960   ❌
-- con_fk:    94
```

El backend deja la FK en NULL y dice en su doc:

> "Para vincular detalles históricos con sus cabeceras, se recomienda realizar el cruce por `documento` y `fecha_emision`"

**Esto es bad design.** La FK existe por algo — el JOIN por documento es lento (no hay índice óptimo) y propenso a confusiones cuando un documento se reutiliza en otro año.

### Acción requerida

1. **Al insertar el detalle, capturar el `id` de la venta recién upserteada** y populrar `vd.venta_id`:
   ```python
   def sync_one_factura(factura_dat_row, detalle_dat_rows):
       # 1. Upsert de la cabecera y obtener su id de Supabase
       venta_record = venta_to_record(factura_dat_row)
       resp = client.table('ventas').upsert(
           venta_record,
           on_conflict='id_unico'
       ).execute()
       venta_id = resp.data[0]['id']  # 🔑 capturar el id real

       # 2. Insertar detalle con la FK ya rellena
       detalle_records = [
           detalle_to_record(d, factura_dat_row['THT_FACTORREFERENCIAL'], venta_id)
           for d in detalle_dat_rows
       ]
       client.table('ventas_detalle').upsert(
           detalle_records,
           on_conflict='documento,codigo_producto'  # o el unique adecuado
       ).execute()
   ```

2. **Backfill para los 52,960 huérfanos existentes:**
   ```sql
   UPDATE ventas_detalle vd
   SET venta_id = v.id
   FROM ventas v
   WHERE vd.venta_id IS NULL
     AND vd.documento = v.documento
     AND v.status = 1;
   -- Validar:
   SELECT COUNT(*) FROM ventas_detalle WHERE venta_id IS NULL;
   -- Esperado: 0 (o muy cerca; podría haber huérfanos legítimos si la cabecera fue anulada)
   ```

3. **Crear índice si no existe:**
   ```sql
   CREATE INDEX IF NOT EXISTS idx_ventas_detalle_venta_id
     ON ventas_detalle(venta_id);
   CREATE INDEX IF NOT EXISTS idx_ventas_documento
     ON ventas(documento);
   ```

## 🐛 Bug #5 — Posibles ventas duplicadas

### Evidencia

```sql
SELECT documento, COUNT(*) FROM ventas WHERE status = 1
GROUP BY documento HAVING COUNT(*) > 1;
-- Devuelve algunos documentos repetidos
```

Vimos las ventas 25189 y 25190 con `total_neto` idéntico — sospechoso.

### Causa raíz probable

El upsert por `id_unico` requiere que `id_unico` sea **realmente único** para cada factura. Si el widget está generando o leyendo mal `id_unico`, dos facturas distintas pueden colisionar y crear filas duplicadas.

### Acción

1. **Verificar que `id_unico` es único en HybridLite** (probablemente `THT_IDÚNICO` o similar). Confirmar que se está leyendo el campo correcto.
2. **Agregar constraint en Supabase para fallar rápido en caso de duplicado:**
   ```sql
   ALTER TABLE ventas ADD CONSTRAINT ventas_id_unico_unique UNIQUE (id_unico);
   ```
   Si esto falla porque ya hay duplicados, primero ejecutar:
   ```sql
   -- Identificar duplicados
   SELECT id_unico, COUNT(*) FROM ventas
   WHERE id_unico IS NOT NULL
   GROUP BY id_unico HAVING COUNT(*) > 1;

   -- Borrar el más reciente cuando hay duplicados (el inserción más vieja es la "real")
   DELETE FROM ventas v
   WHERE v.id IN (
     SELECT id FROM (
       SELECT id, ROW_NUMBER() OVER (PARTITION BY id_unico ORDER BY id ASC) AS rn
       FROM ventas
       WHERE id_unico IS NOT NULL
     ) sub
     WHERE rn > 1
   );
   ```

CREATE OR REPLACE VIEW public.vw_ventas_items_usd AS
SELECT v.id AS venta_id,
  v.fecha_emision,
  v.status,
  vd.codigo_producto,
  CASE
    WHEN vd.cantidad > 1000000::numeric THEN 1.0
    WHEN vd.cantidad <= 0::numeric THEN 0.0
    ELSE vd.cantidad
  END::numeric(20,4) AS cantidad,
  COALESCE(vd.precio_venta, 0::numeric)::numeric(20,4) AS precio_unitario_usd,
  COALESCE(p.costo, 0::numeric)::numeric(20,4) AS costo_unitario_usd,
  (CASE
    WHEN vd.cantidad > 1000000::numeric THEN 1.0
    WHEN vd.cantidad <= 0::numeric THEN 0.0
    ELSE vd.cantidad
  END * COALESCE(vd.precio_venta, 0::numeric))::numeric(20,4) AS subtotal_usd,
  (CASE
    WHEN vd.cantidad > 1000000::numeric THEN 1.0
    WHEN vd.cantidad <= 0::numeric THEN 0.0
    ELSE vd.cantidad
  END * (COALESCE(vd.precio_venta, 0::numeric) - COALESCE(p.costo, 0::numeric)))::numeric(20,4) AS ganancia_item_usd
FROM ventas v
JOIN ventas_detalle vd ON v.id = vd.venta_id   -- ✅ FK ya poblada, vuelve al JOIN clásico
LEFT JOIN productos p ON vd.codigo_producto = p.codigo_interno
WHERE v.status = 1;
```

CREATE OR REPLACE VIEW public.vw_ventas_usd AS
SELECT agg.venta_id,
  agg.fecha_emision,
  agg.status,
  agg.total_usd,
  agg.ganancia_total_usd,
  agg.items_count,
  v.documento,
  v.created_at,
  v.rif_cliente,
  v.total_neto AS original_total_neto_ves,
  v.total_impuesto AS original_total_impuesto_ves,
  COALESCE(v.total_neto, 0::numeric)::numeric(20,4) AS total_neto_usd,
  COALESCE(v.total_bruto, 0::numeric)::numeric(20,4) AS total_bruto_usd,
  COALESCE(v.total_impuesto, 0::numeric)::numeric(20,4) AS total_impuesto_usd,
  COALESCE(c.nombre, 'Cliente Genérico'::text) AS nombre_cliente,
  v.metodo_pago,
  v.id_unico
FROM ( SELECT vw_ventas_items_usd.venta_id,
          vw_ventas_items_usd.fecha_emision,
          vw_ventas_items_usd.status,
          sum(vw_ventas_items_usd.subtotal_usd)::numeric(20,4) AS total_usd,
          sum(vw_ventas_items_usd.ganancia_item_usd)::numeric(20,4) AS ganancia_total_usd,
          sum(vw_ventas_items_usd.cantidad)::numeric(20,4) AS items_count
        FROM vw_ventas_items_usd
        GROUP BY vw_ventas_items_usd.venta_id, vw_ventas_items_usd.fecha_emision, vw_ventas_items_usd.status) agg
JOIN ventas v ON v.id = agg.venta_id
LEFT JOIN LATERAL ( SELECT clientes.nombre FROM clientes
                    WHERE clientes.rif = v.rif_cliente LIMIT 1) c ON true;
```
